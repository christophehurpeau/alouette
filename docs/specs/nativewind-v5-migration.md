# Spec: Migration from Uniwind to NativeWind v5 (deferred)

**Status:** Proposed / deferred — not scheduled. Revisit when NativeWind v5 reaches a stable, production-ready release.
**Current stack:** Uniwind `1.6.3` (Tailwind CSS v4 for React Native).
**Target:** NativeWind v5 (the Tailwind v4-aligned major).

---

## 1. Context & motivation

Alouette currently styles every component with Tailwind classes via **Uniwind**, and its entire design-token system is authored in **Tailwind v4 CSS-first** syntax (`@theme`, `@utility`, `@custom-variant`, `@layer theme`) generated by [build-css.ts](../../packages/alouette/scripts/build-css.ts) into [global.css](../../packages/alouette/src/global.css).

NativeWind is the most widely adopted Tailwind-for-RN library (Expo-aligned, large community, longer track record). The reason this migration is **deferred rather than rejected** is purely a version-compatibility question:

- **NativeWind v4 only supports Tailwind v3.** Migrating today would force a regression off Tailwind v4 and a full rewrite of the token pipeline into a legacy `tailwind.config.js` — not worth it.
- **NativeWind v5 targets Tailwind v4** (CSS-first config, `@theme`, CSS variables, P3 colors, CSS animations). It keeps our authoring model largely intact. As of writing it is **pre-release / not recommended for production**.

Therefore: do not migrate now. This spec captures the plan so the migration can be executed quickly once NativeWind v5 is stable.

> Note: This migration is **not currently justified by a feature need** — the headline NativeWind advantages (free Reanimated animations, large plugin ecosystem) are features Alouette does not use (animations are driven by `react-native-reanimated` directly; safe areas via `react-native-safe-area-context`). The motivation is ecosystem maturity / reduced reliance on a younger library, weighed against migration cost.

## 2. Trigger conditions (when to pick this up)

Start the migration only when **all** of the following hold:

1. NativeWind v5 has a **stable (non-pre-release) release** with documented Tailwind v4 CSS-first support.
2. NativeWind v5 supports **cascading CSS variables / theme classes that override child tokens** on native (our 12-theme semantic cascade depends on this).
3. A documented runtime API exists for **reading a CSS variable value in JS** (replacement for `useCSSVariable`) — or an acceptable workaround is agreed.
4. Vite / Storybook web support is confirmed for our [storybook-native-app](../../packages/storybook-native-app) setup.

If any condition fails, stay on Uniwind and re-evaluate later.

## 3. Current Uniwind surface area (what must be replaced)

### Pervasive (no API change expected)

- `className` on RN components — ~50+ files. Should carry over directly.
- Responsive breakpoints (`sm:`/`md:`/`lg:`/`xl:`) — e.g. [SwitchBreakpoints.tsx](../../packages/alouette/src/windowSize/SwitchBreakpoints.tsx).

### Library-agnostic (no change)

- `tailwind-variants` (`tv()`), `tailwind-merge`, `tailwindcss` v4 — work with NativeWind too.

## 4. Known migration risks & differences

1. **`useCSSVariable` (8 sites)** — the hardest part. These read theme tokens at runtime to feed non-className props (gradient colors, icon color, input placeholder, switch track/thumb). Plan a shared JS token resolver keyed by the active theme so these don't depend on a Uniwind-only hook.
2. **CSS variable resolution on native** — NativeWind transpiles CSS variables to RN's style system. Confirm that our **theme cascade** (child components inherit tokens from the nearest theme scope, no `var()` chains) behaves identically. This is the core of the 12-theme system; prototype it first.
3. **Babel transform** — NativeWind adds `nativewind/babel` (Uniwind is Metro-plugin-only). Expect slower cold builds and an extra moving part.
4. **Theme switching needs a provider/context** — Uniwind switches themes without a React context wrapper; NativeWind v5 uses `VariableContextProvider` (and deprecates v4's `vars()`). [AlouetteProvider.tsx](../../packages/alouette/src/core/AlouetteProvider.tsx) will own this.
5. **Dark mode strategy** — NativeWind v5 favors class-based dark mode; confirm it composes with our `light_*`/`dark_*` semantic classes.
6. **Performance** — Uniwind benchmarks itself ~2–3× faster than NativeWind (vendor numbers; verify against our own Storybook screens if perf matters).

## 5. Execution plan (when triggered)

1. **Spike (timeboxed):** on a branch, install NativeWind v5 in [storybook-native-app](../../packages/storybook-native-app); port **one** semantic theme (e.g. `brand`) end-to-end and **one** `useCSSVariable` consumer ([Icon.tsx](../../packages/alouette/src/ui/primitives/Icon.tsx) or [Switch.tsx](../../packages/alouette/src/ui/inputs/Switch.tsx)). Validate the theme cascade renders identically on iOS, Android, and web. **Go/no-go gate.**
2. **Token pipeline:** adapt [build-css.ts](../../packages/alouette/scripts/build-css.ts) output to NativeWind v5's expected CSS-first format; regenerate [global.css](../../packages/alouette/src/global.css) via `yarn workspace alouette build:css`.
3. **Theming primitives:** rewrite [SemanticScope.tsx](../../packages/alouette/src/ui/containers/SemanticScope.tsx) (replace `ScopedTheme`/`useUniwind`) and theme bootstrapping in [AlouetteProvider.tsx](../../packages/alouette/src/core/AlouetteProvider.tsx) + [AlouetteDecorator.tsx](../../packages/alouette/src/core/AlouetteDecorator.tsx) (replace `Uniwind.setTheme`).
4. **`useCSSVariable` consumers:** introduce a shared token resolver; migrate all 8 sites.
5. **Build config:** metro (`withUniwindConfig` → NativeWind + babel), Vite/web ([vite.config.ts](../../packages/storybook-native-app/vite.config.ts)), and type defs (`uniwind.d.ts` → NativeWind types).
6. **Cleanup:** remove `uniwind` deps from all `package.json`; update [CLAUDE.md](../../CLAUDE.md) and the `uniwind` skill docs.

## 6. Verification

- `yarn tsc`, `yarn lint`, `yarn test` all pass.
- `yarn storybook` runs; every story renders correctly across **all 12 themes** (light/dark × brand/info/success/warning/danger + base) on **iOS, Android, and web**.
- Visual parity check on the screens that use `useCSSVariable` (gradients, Icon, Switch, InputText placeholder).
- `yarn build` produces working dual web (ES) + RN (CJS/ES) output.

## 7. Decision

**Defer.** Stay on Uniwind for now. Execute this spec only when the §2 trigger conditions are met. Migration cost is concentrated in the token pipeline rewrite and the `useCSSVariable` call sites; the benefit is ecosystem maturity rather than any feature Alouette currently needs.
